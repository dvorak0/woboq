31// A simple implementation of N-dimensional dual numbers, for automatically
32// computing exact derivatives of functions.
33//
34// While a complete treatment of the mechanics of automatic differentation is
35// beyond the scope of this header (see
36// http://en.wikipedia.org/wiki/Automatic_differentiation for details), the
37// basic idea is to extend normal arithmetic with an extra element, "e," often
38// denoted with the greek symbol epsilon, such that e != 0 but e^2 = 0. Dual
39// numbers are extensions of the real numbers analogous to complex numbers:
40// whereas complex numbers augment the reals by introducing an imaginary unit i
41// such that i^2 = -1, dual numbers introduce an "infinitesimal" unit e such
42// that e^2 = 0. Dual numbers have two components: the "real" component and the
43// "infinitesimal" component, generally written as x + y*e. Surprisingly, this
44// leads to a convenient method for computing exact derivatives without needing
45// to manipulate complicated symbolic expressions.
46//
47// For example, consider the function
48//
49// f(x) = x^2 ,
50//
51// evaluated at 10. Using normal arithmetic, f(10) = 100, and df/dx(10) = 20.
52// Next, augument 10 with an infinitesimal to get:
53//
54// f(10 + e) = (10 + e)^2
55// = 100 + 2 * 10 * e + e^2
56// = 100 + 20 * e -+-
57// -- |
58// | +--- This is zero, since e^2 = 0
59// |
60// +----------------- This is df/dx!
61//
62// Note that the derivative of f with respect to x is simply the infinitesimal
63// component of the value of f(x + e). So, in order to take the derivative of
64// any function, it is only necessary to replace the numeric "object" used in
65// the function with one extended with infinitesimals. The class Jet, defined in
66// this header, is one such example of this, where substitution is done with
67// templates.
68//
69// To handle derivatives of functions taking multiple arguments, different
70// infinitesimals are used, one for each variable to take the derivative of. For
71// example, consider a scalar function of two scalar parameters x and y:
72//
73// f(x, y) = x^2 + x * y
74//
75// Following the technique above, to compute the derivatives df/dx and df/dy for
76// f(1, 3) involves doing two evaluations of f, the first time replacing x with
77// x + e, the second time replacing y with y + e.
78//
79// For df/dx:
80//
81// f(1 + e, y) = (1 + e)^2 + (1 + e) * 3
82// = 1 + 2 * e + 3 + 3 * e
83// = 4 + 5 * e
84//
85// --> df/dx = 5
86//
87// For df/dy:
88//
89// f(1, 3 + e) = 1^2 + 1 * (3 + e)
90// = 1 + 3 + e
91// = 4 + e
92//
93// --> df/dy = 1
94//
95// To take the gradient of f with the implementation of dual numbers ("jets") in
96// this file, it is necessary to create a single jet type which has components
97// for the derivative in x and y, and passing them to a templated version of f:
98//
99// template<typename T>
100// T f(const T &x, const T &y) {
101// return x * x + x * y;
102// }
103//
104// // The "2" means there should be 2 dual number components.
105// Jet<double, 2> x(0); // Pick the 0th dual number for x.
106// Jet<double, 2> y(1); // Pick the 1st dual number for y.
107// Jet<double, 2> z = f(x, y);
108//
109// LOG(INFO) << "df/dx = " << z.v[0]
110// << "df/dy = " << z.v[1];
111//
112// Most users should not use Jet objects directly; a wrapper around Jet objects,
113// which makes computing the derivative, gradient, or jacobian of templated
114// functors simple, is in autodiff.h. Even autodiff.h should not be used
115// directly; instead autodiff_cost_function.h is typically the file of interest.
116//
117// For the more mathematically inclined, this file implements first-order
118// "jets". A 1st order jet is an element of the ring
119//
120// T[N] = T[t_1, ..., t_N] / (t_1, ..., t_N)^2
121//
122// which essentially means that each jet consists of a "scalar" value 'a' from T
123// and a 1st order perturbation vector 'v' of length N:
124//
125// x = a + \sum_i v[i] t_i
126//
127// A shorthand is to write an element as x = a + u, where u is the pertubation.
128// Then, the main point about the arithmetic of jets is that the product of
129// perturbations is zero:
130//
131// (a + u) * (b + v) = ab + av + bu + uv
132// = ab + (av + bu) + 0
133//
134// which is what operator* implements below. Addition is simpler:
135//
136// (a + u) + (b + v) = (a + b) + (u + v).
137//
138// The only remaining question is how to evaluate the function of a jet, for
139// which we use the chain rule:
140//
141// f(a + u) = f(a) + f'(a) u
142//
143// where f'(a) is the (scalar) derivative of f at a.
144//
145// By pushing these things through sufficiently and suitably templated
146// functions, we can do automatic differentiation. Just be sure to turn on
147// function inlining and common-subexpression elimination, or it will be very
148// slow!
149//
150// WARNING: Most Ceres users should not directly include this file or know the
151// details of how jets work. Instead the suggested method for automatic
152// derivatives is to use autodiff_cost_function.h, which is a wrapper around
153// both jets.h and autodiff.h to make taking derivatives of cost functions for
154// use in Ceres easier.

171template <typename T, int N>
172struct Jet {
173 enum { DIMENSION = N };
174
175 // Default-construct "a" because otherwise this can lead to false errors about
176 // uninitialized uses when other classes relying on default constructed T
177 // (where T is a Jet<T, N>). This usually only happens in opt mode. Note that
178 // the C++ standard mandates that e.g. default constructed doubles are
179 // initialized to 0.0; see sections 8.5 of the C++03 standard.
180 Jet() : a() {
181 v.setZero();
182 }
183
184 // Constructor from scalar: a + 0.
185 explicit Jet(const T& value) {
186 a = value;
187 v.setZero();
188 }
189
190 // Constructor from scalar plus variable: a + t_i.
191 Jet(const T& value, int k) {
192 a = value;
193 v.setZero();
194 v[k] = T(1.0);
195 }
196
197 // Constructor from scalar and vector part
198 // The use of Eigen::DenseBase allows Eigen expressions
199 // to be passed in without being fully evaluated until
200 // they are assigned to v
201 template<typename Derived>
202 EIGEN_STRONG_INLINE Jet(const T& a, const Eigen::DenseBase<Derived> &v)
203 : a(a), v(v) {
204 }
205
206 // Compound operators
207 Jet<T, N>& operator+=(const Jet<T, N> &y) {
208 *this = *this + y;
209 return *this;
210 }
211
212 Jet<T, N>& operator-=(const Jet<T, N> &y) {
213 *this = *this - y;
214 return *this;
215 }
216
217 Jet<T, N>& operator*=(const Jet<T, N> &y) {
218 *this = *this * y;
219 return *this;
220 }
221
222 Jet<T, N>& operator/=(const Jet<T, N> &y) {
223 *this = *this / y;
224 return *this;
225 }
226
227 // The scalar part.
228 T a;
229
230 // The infinitesimal part.
231 //
232 // We allocate Jets on the stack and other places they might not be aligned
233 // to X(=16 [SSE], 32 [AVX] etc)-byte boundaries, which would prevent the safe
234 // use of vectorisation. If we have C++11, we can specify the alignment.
235 // However, the standard gives wide lattitude as to what alignments are valid,
236 // and it might be that the maximum supported alignment *guaranteed* to be
237 // supported is < 16, in which case we do not specify an alignment, as this
238 // implies the host is not a modern x86 machine. If using < C++11, we cannot
239 // specify alignment.
240#ifndef CERES_USE_CXX11
241 // Without >= C++11, we cannot specify the alignment so fall back to safe,
242 // unvectorised version.
243 Eigen::Matrix<T, N, 1, Eigen::DontAlign> v;
244#else
245 // Enable vectorisation iff the maximum supported scalar alignment is >=
246 // 16 bytes, as this is the minimum required by Eigen for any vectorisation.
247 //
248 // NOTE: It might be the case that we could get >= 16-byte alignment even if
249 // kMaxAlignBytes < 16. However we can't guarantee that this
250 // would happen (and it should not for any modern x86 machine) and if it
251 // didn't, we could get misaligned Jets.
252 static constexpr int kAlignOrNot =
253 16 <= ::ceres::port_constants::kMaxAlignBytes
254 ? Eigen::AutoAlign : Eigen::DontAlign;
255#if defined(EIGEN_MAX_ALIGN_BYTES)
256 // Eigen >= 3.3 supports AVX & FMA instructions that require 32-byte alignment
257 // (greater for AVX512). Rather than duplicating the detection logic, use
258 // Eigen's macro for the alignment size.
259 //
260 // NOTE: EIGEN_MAX_ALIGN_BYTES can be > 16 (e.g. 32 for AVX), even though
261 // kMaxAlignBytes will max out at 16. We are therefore relying on
262 // Eigen's detection logic to ensure that this does not result in
263 // misaligned Jets.
264#define CERES_JET_ALIGN_BYTES EIGEN_MAX_ALIGN_BYTES
265#else
266 // Eigen < 3.3 only supported 16-byte alignment.
267#define CERES_JET_ALIGN_BYTES 16
268#endif
269 // Default to the native alignment if 16-byte alignment is not guaranteed to
270 // be supported. We cannot use alignof(T) as if we do, GCC 4.8 complains that
271 // the alignment 'is not an integer constant', although Clang accepts it.
272 static constexpr size_t kAlignment = kAlignOrNot == Eigen::AutoAlign
273 ? CERES_JET_ALIGN_BYTES : alignof(double);
274#undef CERES_JET_ALIGN_BYTES
275 alignas(kAlignment) Eigen::Matrix<T, N, 1, kAlignOrNot> v;
276#endif
277};

351// Binary /
352template<typename T, int N> inline
353Jet<T, N> operator/(const Jet<T, N>& f,
354 const Jet<T, N>& g) {
355 // This uses:
356 //
357 // a + u (a + u)(b - v) (a + u)(b - v)
358 // ----- = -------------- = --------------
359 // b + v (b + v)(b - v) b^2
360 //
361 // which holds because v*v = 0.
362 const T g_a_inverse = T(1.0) / g.a;
363 const T f_a_by_g_a = f.a * g_a_inverse;
364 return Jet<T, N>(f.a * g_a_inverse, (f.v - f_a_by_g_a * g.v) * g_a_inverse);
365}
366
367// Binary / with a scalar: s / x
368template<typename T, int N> inline
369Jet<T, N> operator/(T s, const Jet<T, N>& g) {
370 const T minus_s_g_a_inverse2 = -s / (g.a * g.a);
371 return Jet<T, N>(s / g.a, g.v * minus_s_g_a_inverse2);
372}

374// Binary / with a scalar: x / s
375template<typename T, int N> inline
376Jet<T, N> operator/(const Jet<T, N>& f, T s) {
377 const T s_inverse = T(1.0) / s;
378 return Jet<T, N>(f.a * s_inverse, f.v * s_inverse);
379}

428// In general, f(a + h) ~= f(a) + f'(a) h, via the chain rule.
429
430// abs(x + h) ~= x + h or -(x + h)
431template <typename T, int N> inline
432Jet<T, N> abs(const Jet<T, N>& f) {
433 return f.a < T(0.0) ? -f : f;
434}
435
436// log(a + h) ~= log(a) + h / a
437template <typename T, int N> inline
438Jet<T, N> log(const Jet<T, N>& f) {
439 const T a_inverse = T(1.0) / f.a;
440 return Jet<T, N>(log(f.a), f.v * a_inverse);
441}
442
443// exp(a + h) ~= exp(a) + exp(a) h
444template <typename T, int N> inline
445Jet<T, N> exp(const Jet<T, N>& f) {
446 const T tmp = exp(f.a);
447 return Jet<T, N>(tmp, tmp * f.v);
448}
449
450// sqrt(a + h) ~= sqrt(a) + h / (2 sqrt(a))
451template <typename T, int N> inline
452Jet<T, N> sqrt(const Jet<T, N>& f) {
453 const T tmp = sqrt(f.a);
454 const T two_a_inverse = T(1.0) / (T(2.0) * tmp);
455 return Jet<T, N>(tmp, f.v * two_a_inverse);
456}
457
458// cos(a + h) ~= cos(a) - sin(a) h
459template <typename T, int N> inline
460Jet<T, N> cos(const Jet<T, N>& f) {
461 return Jet<T, N>(cos(f.a), - sin(f.a) * f.v);
462}
463
464// acos(a + h) ~= acos(a) - 1 / sqrt(1 - a^2) h
465template <typename T, int N> inline
466Jet<T, N> acos(const Jet<T, N>& f) {
467 const T tmp = - T(1.0) / sqrt(T(1.0) - f.a * f.a);
468 return Jet<T, N>(acos(f.a), tmp * f.v);
469}
470
471// sin(a + h) ~= sin(a) + cos(a) h
472template <typename T, int N> inline
473Jet<T, N> sin(const Jet<T, N>& f) {
474 return Jet<T, N>(sin(f.a), cos(f.a) * f.v);
475}
476
477// asin(a + h) ~= asin(a) + 1 / sqrt(1 - a^2) h
478template <typename T, int N> inline
479Jet<T, N> asin(const Jet<T, N>& f) {
480 const T tmp = T(1.0) / sqrt(T(1.0) - f.a * f.a);
481 return Jet<T, N>(asin(f.a), tmp * f.v);
482}
483
484// tan(a + h) ~= tan(a) + (1 + tan(a)^2) h
485template <typename T, int N> inline
486Jet<T, N> tan(const Jet<T, N>& f) {
487 const T tan_a = tan(f.a);
488 const T tmp = T(1.0) + tan_a * tan_a;
489 return Jet<T, N>(tan_a, tmp * f.v);
490}
491
492// atan(a + h) ~= atan(a) + 1 / (1 + a^2) h
493template <typename T, int N> inline
494Jet<T, N> atan(const Jet<T, N>& f) {
495 const T tmp = T(1.0) / (T(1.0) + f.a * f.a);
496 return Jet<T, N>(atan(f.a), tmp * f.v);
497}
498
499// sinh(a + h) ~= sinh(a) + cosh(a) h
500template <typename T, int N> inline
501Jet<T, N> sinh(const Jet<T, N>& f) {
502 return Jet<T, N>(sinh(f.a), cosh(f.a) * f.v);
503}
504
505// cosh(a + h) ~= cosh(a) + sinh(a) h
506template <typename T, int N> inline
507Jet<T, N> cosh(const Jet<T, N>& f) {
508 return Jet<T, N>(cosh(f.a), sinh(f.a) * f.v);
509}
510
511// tanh(a + h) ~= tanh(a) + (1 - tanh(a)^2) h
512template <typename T, int N> inline
513Jet<T, N> tanh(const Jet<T, N>& f) {
514 const T tanh_a = tanh(f.a);
515 const T tmp = T(1.0) - tanh_a * tanh_a;
516 return Jet<T, N>(tanh_a, tmp * f.v);
517}

537// Bessel functions of the first kind with integer order equal to 0, 1, n.
538//
539// Microsoft has deprecated the j[0,1,n]() POSIX Bessel functions in favour of
540// _j[0,1,n](). Where available on MSVC, use _j[0,1,n]() to avoid deprecated
541// function errors in client code (the specific warning is suppressed when
542// Ceres itself is built).
543inline double BesselJ0(double x) {
544#if defined(CERES_MSVC_USE_UNDERSCORE_PREFIXED_BESSEL_FUNCTIONS)
545 return _j0(x);
546#else
547 return j0(x);
548#endif
549}
550inline double BesselJ1(double x) {
551#if defined(CERES_MSVC_USE_UNDERSCORE_PREFIXED_BESSEL_FUNCTIONS)
552 return _j1(x);
553#else
554 return j1(x);
555#endif
556}
557inline double BesselJn(int n, double x) {
558#if defined(CERES_MSVC_USE_UNDERSCORE_PREFIXED_BESSEL_FUNCTIONS)
559 return _jn(n, x);
560#else
561 return jn(n, x);
562#endif
563}

565// For the formulae of the derivatives of the Bessel functions see the book:
566// Olver, Lozier, Boisvert, Clark, NIST Handbook of Mathematical Functions,
567// Cambridge University Press 2010.
568//
569// Formulae are also available at http://dlmf.nist.gov
570
571// See formula http://dlmf.nist.gov/10.6#E3
572// j0(a + h) ~= j0(a) - j1(a) h
573template <typename T, int N> inline
574Jet<T, N> BesselJ0(const Jet<T, N>& f) {
575 return Jet<T, N>(BesselJ0(f.a),
576 -BesselJ1(f.a) * f.v);
577}
578
579// See formula http://dlmf.nist.gov/10.6#E1
580// j1(a + h) ~= j1(a) + 0.5 ( j0(a) - j2(a) ) h
581template <typename T, int N> inline
582Jet<T, N> BesselJ1(const Jet<T, N>& f) {
583 return Jet<T, N>(BesselJ1(f.a),
584 T(0.5) * (BesselJ0(f.a) - BesselJn(2, f.a)) * f.v);
585}

587// See formula http://dlmf.nist.gov/10.6#E1
588// j_n(a + h) ~= j_n(a) + 0.5 ( j_{n-1}(a) - j_{n+1}(a) ) h
589template <typename T, int N> inline
590Jet<T, N> BesselJn(int n, const Jet<T, N>& f) {
591 return Jet<T, N>(BesselJn(n, f.a),
592 T(0.5) * (BesselJn(n - 1, f.a) - BesselJn(n + 1, f.a)) * f.v);
593}

595// Jet Classification. It is not clear what the appropriate semantics are for
596// these classifications. This picks that IsFinite and isnormal are "all"
597// operations, i.e. all elements of the jet must be finite for the jet itself
598// to be finite (or normal). For IsNaN and IsInfinite, the answer is less
599// clear. This takes a "any" approach for IsNaN and IsInfinite such that if any
600// part of a jet is nan or inf, then the entire jet is nan or inf. This leads
601// to strange situations like a jet can be both IsInfinite and IsNaN, but in
602// practice the "any" semantics are the most useful for e.g. checking that
603// derivatives are sane.
604
605// The jet is finite if all parts of the jet are finite.
606template <typename T, int N> inline
607bool IsFinite(const Jet<T, N>& f) {
608 if (!IsFinite(f.a)) {
609 return false;
610 }
611 for (int i = 0; i < N; ++i) {
612 if (!IsFinite(f.v[i])) {
613 return false;
614 }
615 }
616 return true;
617}
618
619// The jet is infinite if any part of the jet is infinite.
620template <typename T, int N> inline
621bool IsInfinite(const Jet<T, N>& f) {
622 if (IsInfinite(f.a)) {
623 return true;
624 }
625 for (int i = 0; i < N; i++) {
626 if (IsInfinite(f.v[i])) {
627 return true;
628 }
629 }
630 return false;
631}
632
633// The jet is NaN if any part of the jet is NaN.
634template <typename T, int N> inline
635bool IsNaN(const Jet<T, N>& f) {
636 if (IsNaN(f.a)) {
637 return true;
638 }
639 for (int i = 0; i < N; ++i) {
640 if (IsNaN(f.v[i])) {
641 return true;
642 }
643 }
644 return false;
645}

647// The jet is normal if all parts of the jet are normal.
648template <typename T, int N> inline
649bool IsNormal(const Jet<T, N>& f) {
650 if (!IsNormal(f.a)) {
651 return false;

653 for (int i = 0; i < N; ++i) {

661// atan2(b + db, a + da) ~= atan2(b, a) + (- b da + a db) / (a^2 + b^2)
662//
663// In words: the rate of change of theta is 1/r times the rate of
664// change of (x, y) in the positive angular direction.
665template <typename T, int N> inline
666Jet<T, N> atan2(const Jet<T, N>& g, const Jet<T, N>& f) {
667 // Note order of arguments:
668 //
669 // f = a + da
670 // g = b + db
671
672 T const tmp = T(1.0) / (f.a * f.a + g.a * g.a);
673 return Jet<T, N>(atan2(g.a, f.a), tmp * (- g.a * f.v + f.a * g.v));
674}

677// pow -- base is a differentiable function, exponent is a constant.
678// (a+da)^p ~= a^p + p*a^(p-1) da
679template <typename T, int N> inline
680Jet<T, N> pow(const Jet<T, N>& f, double g) {
681 T const tmp = g * pow(f.a, g - T(1.0));
682 return Jet<T, N>(pow(f.a, g), tmp * f.v);
683}
684
685// pow -- base is a constant, exponent is a differentiable function.
686// We have various special cases, see the comment for pow(Jet, Jet) for
687// analysis:
688//
689// 1. For f > 0 we have: (f)^(g + dg) ~= f^g + f^g log(f) dg
690//
691// 2. For f == 0 and g > 0 we have: (f)^(g + dg) ~= f^g
692//
693// 3. For f < 0 and integer g we have: (f)^(g + dg) ~= f^g but if dg
694// != 0, the derivatives are not defined and we return NaN.
695
696template <typename T, int N> inline
697Jet<T, N> pow(double f, const Jet<T, N>& g) {
698 if (f == 0 && g.a > 0) {
699 // Handle case 2.
700 return Jet<T, N>(T(0.0));
701 }
702 if (f < 0 && g.a == floor(g.a)) {
703 // Handle case 3.
704 Jet<T, N> ret(pow(f, g.a));
705 for (int i = 0; i < N; i++) {
706 if (g.v[i] != T(0.0)) {
707 // Return a NaN when g.v != 0.
708 ret.v[i] = std::numeric_limits<T>::quiet_NaN();
709 }
710 }
711 return ret;
712 }
713 // Handle case 1.
714 T const tmp = pow(f, g.a);
715 return Jet<T, N>(tmp, log(f) * tmp * g.v);
716}
717
718// pow -- both base and exponent are differentiable functions. This has a
719// variety of special cases that require careful handling.
720//
721// 1. For f > 0:
722// (f + df)^(g + dg) ~= f^g + f^(g - 1) * (g * df + f * log(f) * dg)
723// The numerical evaluation of f * log(f) for f > 0 is well behaved, even for
724// extremely small values (e.g. 1e-99).
725//
726// 2. For f == 0 and g > 1: (f + df)^(g + dg) ~= 0
727// This cases is needed because log(0) can not be evaluated in the f > 0
728// expression. However the function f*log(f) is well behaved around f == 0
729// and its limit as f-->0 is zero.
730//
731// 3. For f == 0 and g == 1: (f + df)^(g + dg) ~= 0 + df
732//
733// 4. For f == 0 and 0 < g < 1: The value is finite but the derivatives are not.
734//
735// 5. For f == 0 and g < 0: The value and derivatives of f^g are not finite.
736//
737// 6. For f == 0 and g == 0: The C standard incorrectly defines 0^0 to be 1
738// "because there are applications that can exploit this definition". We
739// (arbitrarily) decree that derivatives here will be nonfinite, since that
740// is consistent with the behavior for f == 0, g < 0 and 0 < g < 1.
741// Practically any definition could have been justified because mathematical
742// consistency has been lost at this point.
743//
744// 7. For f < 0, g integer, dg == 0: (f + df)^(g + dg) ~= f^g + g * f^(g - 1) df
745// This is equivalent to the case where f is a differentiable function and g
746// is a constant (to first order).
747//
748// 8. For f < 0, g integer, dg != 0: The value is finite but the derivatives are
749// not, because any change in the value of g moves us away from the point
750// with a real-valued answer into the region with complex-valued answers.
751//
752// 9. For f < 0, g noninteger: The value and derivatives of f^g are not finite.
753
754template <typename T, int N> inline
755Jet<T, N> pow(const Jet<T, N>& f, const Jet<T, N>& g) {
756 if (f.a == 0 && g.a >= 1) {
757 // Handle cases 2 and 3.
758 if (g.a > 1) {
759 return Jet<T, N>(T(0.0));
760 }
761 return f;
762 }
763 if (f.a < 0 && g.a == floor(g.a)) {
764 // Handle cases 7 and 8.
765 T const tmp = g.a * pow(f.a, g.a - T(1.0));
766 Jet<T, N> ret(pow(f.a, g.a), tmp * f.v);
767 for (int i = 0; i < N; i++) {
768 if (g.v[i] != T(0.0)) {
769 // Return a NaN when g.v != 0.
770 ret.v[i] = std::numeric_limits<T>::quiet_NaN();
771 }
772 }
773 return ret;
774 }
775 // Handle the remaining cases. For cases 4,5,6,9 we allow the log() function
776 // to generate -HUGE_VAL or NaN, since those cases result in a nonfinite
777 // derivative.
778 T const tmp1 = pow(f.a, g.a);
779 T const tmp2 = g.a * pow(f.a, g.a - T(1.0));
780 T const tmp3 = tmp1 * log(f.a);
781 return Jet<T, N>(tmp1, tmp2 * f.v + tmp3 * g.v);
782}
