13/* CHOLMOD Core module: basic CHOLMOD objects and routines.
14 * Required by all CHOLMOD modules. Requires no other module or package.
15 *
16 * The CHOLMOD modules are:
17 *
18 * Core basic data structures and definitions
19 * Check check/print the 5 CHOLMOD objects, & 3 types of integer vectors
20 * Cholesky sparse Cholesky factorization
21 * Modify sparse Cholesky update/downdate/row-add/row-delete
22 * MatrixOps sparse matrix functions (add, multiply, norm, ...)
23 * Supernodal supernodal sparse Cholesky factorization
24 * Partition graph-partitioning based orderings
25 *
26 * The CHOLMOD objects:
27 * --------------------
28 *
29 * cholmod_common parameters, statistics, and workspace
30 * cholmod_sparse a sparse matrix in compressed column form
31 * cholmod_factor an LL' or LDL' factorization
32 * cholmod_dense a dense matrix
33 * cholmod_triplet a sparse matrix in "triplet" form
34 *
35 * The Core module described here defines the CHOLMOD data structures, and
36 * basic operations on them. To create and solve a sparse linear system Ax=b,
37 * the user must create A and b, populate them with values, and then pass them
38 * to the routines in the CHOLMOD Cholesky module. There are two primary
39 * methods for creating A: (1) allocate space for a column-oriented sparse
40 * matrix and fill it with pattern and values, or (2) create a triplet form
41 * matrix and convert it to a sparse matrix. The latter option is simpler.
42 *
43 * The matrices b and x are typically dense matrices, but can also be sparse.
44 * You can allocate and free them as dense matrices with the
45 * cholmod_allocate_dense and cholmod_free_dense routines.
46 *
47 * The cholmod_factor object contains the symbolic and numeric LL' or LDL'
48 * factorization of sparse symmetric matrix. The matrix must be positive
49 * definite for an LL' factorization. It need only be symmetric and have well-
50 * conditioned leading submatrices for it to have an LDL' factorization
51 * (CHOLMOD does not pivot for numerical stability). It is typically created
52 * with the cholmod_factorize routine in the Cholesky module, but can also
53 * be initialized to L=D=I in the Core module and then modified by the Modify
54 * module. It must be freed with cholmod_free_factor, defined below.
55 *
56 * The Core routines for each object are described below. Each list is split
57 * into two parts: the primary routines and secondary routines.
58 *
59 * ============================================================================
60 * === cholmod_common =========================================================
61 * ============================================================================
62 *
63 * The Common object contains control parameters, statistics, and
64 * You must call cholmod_start before calling any other CHOLMOD routine, and
65 * must call cholmod_finish as your last call to CHOLMOD, with two exceptions:
66 * you may call cholmod_print_common and cholmod_check_common in the Check
67 * module after calling cholmod_finish.
68 *
69 * cholmod_start first call to CHOLMOD
70 * cholmod_finish last call to CHOLMOD
71 * -----------------------------
72 * cholmod_defaults restore default parameters
73 * cholmod_maxrank maximum rank for update/downdate
74 * cholmod_allocate_work allocate workspace in Common
75 * cholmod_free_work free workspace in Common
76 * cholmod_clear_flag clear Flag workspace in Common
77 * cholmod_error called when CHOLMOD encounters an error
78 * cholmod_dbound for internal use in CHOLMOD only
79 * cholmod_hypot compute sqrt (x*x + y*y) accurately
80 * cholmod_divcomplex complex division, c = a/b
81 *
82 * ============================================================================
83 * === cholmod_sparse =========================================================
84 * ============================================================================
85 *
86 * A sparse matrix is held in compressed column form. In the basic type
87 * ("packed", which corresponds to a MATLAB sparse matrix), an n-by-n matrix
88 * with nz entries is held in three arrays: p of size n+1, i of size nz, and x
89 * of size nz. Row indices of column j are held in i [p [j] ... p [j+1]-1] and
90 * in the same locations in x. There may be no duplicate entries in a column.
91 * Row indices in each column may be sorted or unsorted (CHOLMOD keeps track).
92 * A->stype determines the storage mode: 0 if both upper/lower parts are stored,
93 * -1 if A is symmetric and just tril(A) is stored, +1 if symmetric and triu(A)
94 * is stored.
95 *
96 * cholmod_allocate_sparse allocate a sparse matrix
97 * cholmod_free_sparse free a sparse matrix
98 * -----------------------------
99 * cholmod_reallocate_sparse change the size (# entries) of sparse matrix
100 * cholmod_nnz number of nonzeros in a sparse matrix
101 * cholmod_speye sparse identity matrix
102 * cholmod_spzeros sparse zero matrix
103 * cholmod_transpose transpose a sparse matrix
104 * cholmod_ptranspose transpose/permute a sparse matrix
105 * cholmod_transpose_unsym transpose/permute an unsymmetric sparse matrix
106 * cholmod_transpose_sym transpose/permute a symmetric sparse matrix
107 * cholmod_sort sort row indices in each column of sparse matrix
108 * cholmod_band C = tril (triu (A,k1), k2)
109 * cholmod_band_inplace A = tril (triu (A,k1), k2)
110 * cholmod_aat C = A*A'
111 * cholmod_copy_sparse C = A, create an exact copy of a sparse matrix
112 * cholmod_copy C = A, with possible change of stype
113 * cholmod_add C = alpha*A + beta*B
114 * cholmod_sparse_xtype change the xtype of a sparse matrix
115 *
116 * ============================================================================
117 * === cholmod_factor =========================================================
118 * ============================================================================
119 *
120 * The data structure for an LL' or LDL' factorization is too complex to
121 * describe in one sentence. This object can hold the symbolic analysis alone,
122 * or in combination with a "simplicial" (similar to a sparse matrix) or
123 * "supernodal" form of the numerical factorization. Only the routine to free
124 * a factor is primary, since a factor object is created by the factorization
125 * routine (cholmod_factorize). It must be freed with cholmod_free_factor.
126 *
127 * cholmod_free_factor free a factor
128 * -----------------------------
129 * cholmod_allocate_factor allocate a factor (LL' or LDL')
130 * cholmod_reallocate_factor change the # entries in a factor
131 * cholmod_change_factor change the type of factor (e.g., LDL' to LL')
132 * cholmod_pack_factor pack the columns of a factor
133 * cholmod_reallocate_column resize a single column of a factor
134 * cholmod_factor_to_sparse create a sparse matrix copy of a factor
135 * cholmod_copy_factor create a copy of a factor
136 * cholmod_factor_xtype change the xtype of a factor
137 *
138 * Note that there is no cholmod_sparse_to_factor routine to create a factor
139 * as a copy of a sparse matrix. It could be done, after a fashion, but a
140 * lower triangular sparse matrix would not necessarily have a chordal graph,
141 * which would break the many CHOLMOD routines that rely on this property.
142 *
143 * ============================================================================
144 * === cholmod_dense ==========================================================
145 * ============================================================================
146 *
147 * The solve routines and some of the MatrixOps and Modify routines use dense
148 * matrices as inputs. These are held in column-major order. With a leading
149 * dimension of d, the entry in row i and column j is held in x [i+j*d].
150 *
151 * cholmod_allocate_dense allocate a dense matrix
152 * cholmod_free_dense free a dense matrix
153 * -----------------------------
154 * cholmod_zeros allocate a dense matrix of all zeros
155 * cholmod_ones allocate a dense matrix of all ones
156 * cholmod_eye allocate a dense identity matrix
157 * cholmod_sparse_to_dense create a dense matrix copy of a sparse matrix
158 * cholmod_dense_to_sparse create a sparse matrix copy of a dense matrix
159 * cholmod_copy_dense create a copy of a dense matrix
160 * cholmod_copy_dense2 copy a dense matrix (pre-allocated)
161 * cholmod_dense_xtype change the xtype of a dense matrix
162 * cholmod_ensure_dense ensure a dense matrix has a given size and type
163 *
164 * ============================================================================
165 * === cholmod_triplet ========================================================
166 * ============================================================================
167 *
168 * A sparse matrix held in triplet form is the simplest one for a user to
169 * create. It consists of a list of nz entries in arbitrary order, held in
170 * three arrays: i, j, and x, each of length nk. The kth entry is in row i[k],
171 * column j[k], with value x[k]. There may be duplicate values; if A(i,j)
172 * appears more than once, its value is the sum of the entries with those row
173 * and column indices.
174 *
175 * cholmod_allocate_triplet allocate a triplet matrix
176 * cholmod_triplet_to_sparse create a sparse matrix copy of a triplet matrix
177 * cholmod_free_triplet free a triplet matrix
178 * -----------------------------
179 * cholmod_reallocate_triplet change the # of entries in a triplet matrix
180 * cholmod_sparse_to_triplet create a triplet matrix copy of a sparse matrix
181 * cholmod_copy_triplet create a copy of a triplet matrix
182 * cholmod_triplet_xtype change the xtype of a triplet matrix
183 *
184 * ============================================================================
185 * === memory management ======================================================
186 * ============================================================================
187 *
188 * cholmod_malloc malloc wrapper
189 * cholmod_calloc calloc wrapper
190 * cholmod_free free wrapper
191 * cholmod_realloc realloc wrapper
192 * cholmod_realloc_multiple realloc wrapper for multiple objects
193 *
194 * ============================================================================
195 * === Core CHOLMOD prototypes ================================================
196 * ============================================================================
197 *
198 * All CHOLMOD routines (in all modules) use the following protocol for return
199 * values, with one exception:
200 *
201 * int TRUE (1) if successful, or FALSE (0) otherwise.
202 * (exception: cholmod_divcomplex)
203 * SuiteSparse_long a value >= 0 if successful, or -1 otherwise.
204 * double a value >= 0 if successful, or -1 otherwise.
205 * size_t a value > 0 if successful, or 0 otherwise.
206 * void * a non-NULL pointer to newly allocated memory if
207 * successful, or NULL otherwise.
208 * cholmod_sparse * a non-NULL pointer to a newly allocated matrix
209 * if successful, or NULL otherwise.
210 * cholmod_factor * a non-NULL pointer to a newly allocated factor
211 * if successful, or NULL otherwise.
212 * cholmod_triplet * a non-NULL pointer to a newly allocated triplet
213 * matrix if successful, or NULL otherwise.
214 * cholmod_dense * a non-NULL pointer to a newly allocated triplet
215 * matrix if successful, or NULL otherwise.
216 *
217 * The last parameter to all routines is always a pointer to the CHOLMOD
218 * Common object.
219 *
220 * TRUE and FALSE are not defined here, since they may conflict with the user
221 * program. A routine that described here returning TRUE or FALSE returns 1
222 * or 0, respectively. Any TRUE/FALSE parameter is true if nonzero, false if
223 * zero.
224 */

319/* xtype defines the kind of numerical values used: */
320#define CHOLMOD_PATTERN 0 /* pattern only, no numerical values */
321#define CHOLMOD_REAL 1 /* a real matrix */
322#define CHOLMOD_COMPLEX 2 /* a complex matrix (ANSI C99 compatible) */
323#define CHOLMOD_ZOMPLEX 3 /* a complex matrix (MATLAB compatible) */

325/* The xtype of all parameters for all CHOLMOD routines must match.
326 *
327 * CHOLMOD_PATTERN: x and z are ignored.
328 * CHOLMOD_DOUBLE: x is non-null of size nzmax, z is ignored.
329 * CHOLMOD_COMPLEX: x is non-null of size 2*nzmax doubles, z is ignored.
330 * CHOLMOD_ZOMPLEX: x and z are non-null of size nzmax
331 *
332 * In the real case, z is ignored. The kth entry in the matrix is x [k].
333 * There are two methods for the complex case. In the ANSI C99-compatible
334 * CHOLMOD_COMPLEX case, the real and imaginary parts of the kth entry
335 * are in x [2*k] and x [2*k+1], respectively. z is ignored. In the
336 * MATLAB-compatible CHOLMOD_ZOMPLEX case, the real and imaginary
337 * parts of the kth entry are in x [k] and z [k].
338 *
339 * Scalar floating-point values are always passed as double arrays of size 2
340 * (real and imaginary parts). The imaginary part of a scalar is ignored if
341 * the routine operates on a real matrix.
342 *
343 * These Modules support complex and zomplex matrices, with a few exceptions:
344 *
345 * Check all routines
346 * Cholesky all routines
347 * Core all except cholmod_aat, add, band, copy
348 * Demo all routines
349 * Partition all routines
350 * Supernodal all routines support any real, complex, or zomplex input.
351 * There will never be a supernodal zomplex L; a complex
352 * supernodal L is created if A is zomplex.
353 * Tcov all routines
354 * Valgrind all routines
355 *
356 * These Modules provide partial support for complex and zomplex matrices:
357 *
358 * MATLAB all routines support real and zomplex only, not complex,
359 * with the exception of ldlupdate, which supports
360 * real matrices only. This is a minor constraint since
361 * MATLAB's matrices are all real or zomplex.
362 * MatrixOps only norm_dense, norm_sparse, and sdmult support complex
363 * and zomplex
364 *
365 * These Modules do not support complex and zomplex matrices at all:
366 *
367 * Modify all routines support real matrices only
368 */

409 /* parameters for symbolic/numeric factorization and update/downdate */

412 double dbound ; /* Smallest absolute value of diagonal entries of D
413 * for LDL' factorization and update/downdate/rowadd/
414 * rowdel, or the diagonal of L for an LL' factorization.
415 * Entries in the range 0 to dbound are replaced with dbound.
416 * Entries in the range -dbound to 0 are replaced with -dbound. No
417 * changes are made to the diagonal if dbound <= 0. Default: zero */
418
419 double grow0 ; /* For a simplicial factorization, L->i and L->x can
420 * grow if necessary. grow0 is the factor by which
421 * it grows. For the initial space, L is of size MAX (1,grow0) times
422 * the required space. If L runs out of space, the new size of L is
423 * MAX(1.2,grow0) times the new required space. If you do not plan on
424 * modifying the LDL' factorization in the Modify module, set grow0 to
425 * zero (or set grow2 to 0, see below). Default: 1.2 */
426
427 double grow1 ;
428
429 size_t grow2 ; /* For a simplicial factorization, each column j of L
430 * is initialized with space equal to
431 * grow1*L->ColCount[j] + grow2. If grow0 < 1, grow1 < 1, or grow2 == 0,
432 * then the space allocated is exactly equal to L->ColCount[j]. If the
433 * column j runs out of space, it increases to grow1*need + grow2 in
434 * size, where need is the total # of nonzeros in that column. If you do
435 * not plan on modifying the factorization in the Modify module, set
436 * grow2 to zero. Default: grow1 = 1.2, grow2 = 5. */
437
438 size_t maxrank ; /* rank of maximum update/downdate. Valid values:
439 * 2, 4, or 8. A value < 2 is set to 2, and a
440 * value > 8 is set to 8. It is then rounded up to the next highest
441 * power of 2, if not already a power of 2. Workspace (Xwork, below) of
442 * size nrow-by-maxrank double's is allocated for the update/downdate.
443 * If an update/downdate of rank-k is requested, with k > maxrank,
444 * it is done in steps of maxrank. Default: 8, which is fastest.
445 * Memory usage can be reduced by setting maxrank to 2 or 4.
446 */
447
448 double supernodal_switch ; /* supernodal vs simplicial factorization */
449 int supernodal ; /* If Common->supernodal <= CHOLMOD_SIMPLICIAL
450 * (0) then cholmod_analyze performs a
451 * simplicial analysis. If >= CHOLMOD_SUPERNODAL (2), then a supernodal
452 * analysis is performed. If == CHOLMOD_AUTO (1) and
453 * flop/nnz(L) < Common->supernodal_switch, then a simplicial analysis
454 * is done. A supernodal analysis done otherwise.
455 * Default: CHOLMOD_AUTO. Default supernodal_switch = 40 */
456
457 int final_asis ; /* If TRUE, then ignore the other final_* parameters
458 * (except for final_pack).
459 * The factor is left as-is when done. Default: TRUE.*/
460
461 int final_super ; /* If TRUE, leave a factor in supernodal form when
462 * supernodal factorization is finished. If FALSE,
463 * then convert to a simplicial factor when done.
464 * Default: TRUE */
465
466 int final_ll ; /* If TRUE, leave factor in LL' form when done.
467 * Otherwise, leave in LDL' form. Default: FALSE */
468
469 int final_pack ; /* If TRUE, pack the columns when done. If TRUE, and
470 * cholmod_factorize is called with a symbolic L, L is
471 * allocated with exactly the space required, using L->ColCount. If you
472 * plan on modifying the factorization, set Common->final_pack to FALSE,
473 * and each column will be given a little extra slack space for future
474 * growth in fill-in due to updates. Default: TRUE */
475
476 int final_monotonic ; /* If TRUE, ensure columns are monotonic when done.
477 * Default: TRUE */
478
479 int final_resymbol ;/* if cholmod_factorize performed a supernodal
480 * factorization, final_resymbol is true, and
481 * final_super is FALSE (convert a simplicial numeric factorization),
482 * then numerically zero entries that resulted from relaxed supernodal
483 * amalgamation are removed. This does not remove entries that are zero
484 * due to exact numeric cancellation, since doing so would break the
485 * update/downdate rowadd/rowdel routines. Default: FALSE. */
486
487 /* supernodal relaxed amalgamation parameters: */
488 double zrelax [3] ;
489 size_t nrelax [3] ;
490
491 /* Let ns be the total number of columns in two adjacent supernodes.
492 * Let z be the fraction of zero entries in the two supernodes if they
493 * are merged (z includes zero entries from prior amalgamations). The
494 * two supernodes are merged if:
495 * (ns <= nrelax [0]) || (no new zero entries added) ||
496 * (ns <= nrelax [1] && z < zrelax [0]) ||
497 * (ns <= nrelax [2] && z < zrelax [1]) || (z < zrelax [2])
498 *
499 * Default parameters result in the following rule:
500 * (ns <= 4) || (no new zero entries added) ||
501 * (ns <= 16 && z < 0.8) || (ns <= 48 && z < 0.1) || (z < 0.05)
502 */
503
504 int prefer_zomplex ; /* X = cholmod_solve (sys, L, B, Common) computes
505 * x=A\b or solves a related system. If L and B are
506 * both real, then X is real. Otherwise, X is returned as
507 * CHOLMOD_COMPLEX if Common->prefer_zomplex is FALSE, or
508 * CHOLMOD_ZOMPLEX if Common->prefer_zomplex is TRUE. This parameter
509 * is needed because there is no supernodal zomplex L. Suppose the
510 * caller wants all complex matrices to be stored in zomplex form
511 * (MATLAB, for example). A supernodal L is returned in complex form
512 * if A is zomplex. B can be real, and thus X = cholmod_solve (L,B)
513 * should return X as zomplex. This cannot be inferred from the input
514 * arguments L and B. Default: FALSE, since all data types are
515 * supported in CHOLMOD_COMPLEX form and since this is the native type
516 * of LAPACK and the BLAS. Note that the MATLAB/cholmod.c mexFunction
517 * sets this parameter to TRUE, since MATLAB matrices are in
518 * CHOLMOD_ZOMPLEX form.
519 */
520
521 int prefer_upper ; /* cholmod_analyze and cholmod_factorize work
522 * fastest when a symmetric matrix is stored in
523 * upper triangular form when a fill-reducing ordering is used. In
524 * MATLAB, this corresponds to how x=A\b works. When the matrix is
525 * ordered as-is, they work fastest when a symmetric matrix is in lower
526 * triangular form. In MATLAB, R=chol(A) does the opposite. This
527 * parameter affects only how cholmod_read returns a symmetric matrix.
528 * If TRUE (the default case), a symmetric matrix is always returned in
529 * upper-triangular form (A->stype = 1). */

533 * the matrix is not positive definite. Default: FALSE. */

561 /* The cholmod_analyze routine can try many different orderings and select
562 * the best one. It can also try one ordering method multiple times, with
563 * different parameter settings. The default is to use three orderings,
564 * the user's permutation (if provided), AMD which is the fastest ordering
565 * and generally gives good fill-in, and METIS. CHOLMOD's nested dissection
566 * (METIS with a constrained AMD) usually gives a better ordering than METIS
567 * alone (by about 5% to 10%) but it takes more time.
568 *
569 * If you know the method that is best for your matrix, set Common->nmethods
570 * to 1 and set Common->method [0] to the set of parameters for that method.
571 * If you set it to 1 and do not provide a permutation, then only AMD will

578 * want to try. The suite of default methods and their parameters is
579 * described in the cholmod_defaults routine, and summarized here:
580 *

582 * i = 0: user-provided ordering (cholmod_analyze_p only)
583 * i = 1: AMD (for both A and A*A')
584 * i = 2: METIS
585 * i = 3: CHOLMOD's nested dissection (NESDIS), default parameters
586 * i = 4: natural

588 * i = 6: NESDIS with nd_small = 4, no constrained minimum degree
589 * i = 7: NESDIS with no dense node removal

593 * Common.method [...] after calling cholmod_start or cholmod_defaults.

609 * good permutation. In this case, try setting Common->nmethods to 9.

1552/* A symbolic and numeric factorization, either simplicial or supernodal.
1553 * In all cases, the row indices in the columns of L are kept sorted. */
1554
1555typedef struct cholmod_factor_struct
1556{
1557 /* ---------------------------------------------------------------------- */
1558 /* for both simplicial and supernodal factorizations */
1559 /* ---------------------------------------------------------------------- */
1560
1561 size_t n ; /* L is n-by-n */
1562
1563 size_t minor ; /* If the factorization failed, L->minor is the column
1564 * at which it failed (in the range 0 to n-1). A value
1565 * of n means the factorization was successful or
1566 * the matrix has not yet been factorized. */
1567
1568 /* ---------------------------------------------------------------------- */
1569 /* symbolic ordering and analysis */
1570 /* ---------------------------------------------------------------------- */
1571
1572 void *Perm ; /* size n, permutation used */
1573 void *ColCount ; /* size n, column counts for simplicial L */
1574
1575 void *IPerm ; /* size n, inverse permutation. Only created by
1576 * cholmod_solve2 if Bset is used. */
1577
1578 /* ---------------------------------------------------------------------- */
1579 /* simplicial factorization */
1580 /* ---------------------------------------------------------------------- */
1581
1582 size_t nzmax ; /* size of i and x */
1583
1584 void *p ; /* p [0..ncol], the column pointers */
1585 void *i ; /* i [0..nzmax-1], the row indices */
1586 void *x ; /* x [0..nzmax-1], the numerical values */
1587 void *z ;
1588 void *nz ; /* nz [0..ncol-1], the # of nonzeros in each column.
1589 * i [p [j] ... p [j]+nz[j]-1] contains the row indices,
1590 * and the numerical values are in the same locatins
1591 * in x. The value of i [p [k]] is always k. */
1592
1593 void *next ; /* size ncol+2. next [j] is the next column in i/x */
1594 void *prev ; /* size ncol+2. prev [j] is the prior column in i/x.
1595 * head of the list is ncol+1, and the tail is ncol. */
1596
1597 /* ---------------------------------------------------------------------- */
1598 /* supernodal factorization */
1599 /* ---------------------------------------------------------------------- */
1600
1601 /* Note that L->x is shared with the simplicial data structure. L->x has
1602 * size L->nzmax for a simplicial factor, and size L->xsize for a supernodal
1603 * factor. */
1604
1605 size_t nsuper ; /* number of supernodes */
1606 size_t ssize ; /* size of s, integer part of supernodes */
1607 size_t xsize ; /* size of x, real part of supernodes */
1608 size_t maxcsize ; /* size of largest update matrix */
1609 size_t maxesize ; /* max # of rows in supernodes, excl. triangular part */
1610
1611 void *super ; /* size nsuper+1, first col in each supernode */
1612 void *pi ; /* size nsuper+1, pointers to integer patterns */
1613 void *px ; /* size nsuper+1, pointers to real parts */
1614 void *s ; /* size ssize, integer part of supernodes */
1615

1617 /* factorization type */
1618 /* ---------------------------------------------------------------------- */
1619
1620 int ordering ; /* ordering method used */
1621
1622 int is_ll ; /* TRUE if LL', FALSE if LDL' */
1623 int is_super ; /* TRUE if supernodal, FALSE if simplicial */
1624 int is_monotonic ; /* TRUE if columns of L appear in order 0..n-1.
1625 * Only applicable to simplicial numeric types. */
1626
1627 /* There are 8 types of factor objects that cholmod_factor can represent
1628 * (only 6 are used):
1629 *
1630 * Numeric types (xtype is not CHOLMOD_PATTERN)
1631 * --------------------------------------------
1632 *
1633 * simplicial LDL': (is_ll FALSE, is_super FALSE). Stored in compressed
1634 * column form, using the simplicial components above (nzmax, p, i,
1635 * x, z, nz, next, and prev). The unit diagonal of L is not stored,
1636 * and D is stored in its place. There are no supernodes.
1637 *
1638 * simplicial LL': (is_ll TRUE, is_super FALSE). Uses the same storage
1639 * scheme as the simplicial LDL', except that D does not appear.
1640 * The first entry of each column of L is the diagonal entry of
1641 * that column of L.
1642 *
1643 * supernodal LDL': (is_ll FALSE, is_super TRUE). Not used.
1644 * FUTURE WORK: add support for supernodal LDL'
1645 *
1646 * supernodal LL': (is_ll TRUE, is_super TRUE). A supernodal factor,
1647 * using the supernodal components described above (nsuper, ssize,
1648 * xsize, maxcsize, maxesize, super, pi, px, s, x, and z).
1649 *
1650 *
1651 * Symbolic types (xtype is CHOLMOD_PATTERN)
1652 * -----------------------------------------
1653 *
1654 * simplicial LDL': (is_ll FALSE, is_super FALSE). Nothing is present
1655 * except Perm and ColCount.
1656 *
1657 * simplicial LL': (is_ll TRUE, is_super FALSE). Identical to the
1658 * simplicial LDL', except for the is_ll flag.
1659 *
1660 * supernodal LDL': (is_ll FALSE, is_super TRUE). Not used.
1661 * FUTURE WORK: add support for supernodal LDL'
1662 *
1663 * supernodal LL': (is_ll TRUE, is_super TRUE). A supernodal symbolic
1664 * factorization. The simplicial symbolic information is present
1665 * (Perm and ColCount), as is all of the supernodal factorization
1666 * except for the numerical values (x and z).
1667 */
1668
1669 int itype ; /* The integer arrays are Perm, ColCount, p, i, nz,
1670 * next, prev, super, pi, px, and s. If itype is
1671 * CHOLMOD_INT, all of these are int arrays.
1672 * CHOLMOD_INTLONG: p, pi, px are SuiteSparse_long, others int.
1673 * CHOLMOD_LONG: all integer arrays are SuiteSparse_long. */
1674 int xtype ; /* pattern, real, complex, or zomplex */
1675 int dtype ; /* x and z double or float */
1676
1677} cholmod_factor ;

2032/* A sparse matrix stored in triplet form. */

2034typedef struct cholmod_triplet_struct
2035{
2036 size_t nrow ; /* the matrix is nrow-by-ncol */
2037 size_t ncol ;
2038 size_t nzmax ; /* maximum number of entries in the matrix */
2039 size_t nnz ; /* number of nonzeros in the matrix */
2040
2041 void *i ; /* i [0..nzmax-1], the row indices */
2042 void *j ; /* j [0..nzmax-1], the column indices */
2043 void *x ; /* size nzmax or 2*nzmax, if present */
2044 void *z ; /* size nzmax, if present */
2045
2046 int stype ; /* Describes what parts of the matrix are considered:
2047 *
2048 * 0: matrix is "unsymmetric": use both upper and lower triangular parts
2049 * (the matrix may actually be symmetric in pattern and value, but
2050 * both parts are explicitly stored and used). May be square or
2051 * rectangular.
2052 * >0: matrix is square and symmetric. Entries in the lower triangular
2053 * part are transposed and added to the upper triangular part when
2054 * the matrix is converted to cholmod_sparse form.
2055 * <0: matrix is square and symmetric. Entries in the upper triangular
2056 * part are transposed and added to the lower triangular part when
2057 * the matrix is converted to cholmod_sparse form.
2058 *
2059 * Note that stype>0 and stype<0 are different for cholmod_sparse and
2060 * cholmod_triplet. The reason is simple. You can permute a symmetric
2061 * triplet matrix by simply replacing a row and column index with their
2062 * new row and column indices, via an inverse permutation. Suppose
2063 * P = L->Perm is your permutation, and Pinv is an array of size n.
2064 * Suppose a symmetric matrix A is represent by a triplet matrix T, with
2065 * entries only in the upper triangular part. Then the following code:
2066 *
2067 * Ti = T->i ;
2068 * Tj = T->j ;
2069 * for (k = 0 ; k < n ; k++) Pinv [P [k]] = k ;
2070 * for (k = 0 ; k < nz ; k++) Ti [k] = Pinv [Ti [k]] ;
2071 * for (k = 0 ; k < nz ; k++) Tj [k] = Pinv [Tj [k]] ;
2072 *
2073 * creates the triplet form of C=P*A*P'. However, if T initially
2074 * contains just the upper triangular entries (T->stype = 1), after
2075 * permutation it has entries in both the upper and lower triangular
2076 * parts. These entries should be transposed when constructing the
2077 * cholmod_sparse form of A, which is what cholmod_triplet_to_sparse
2078 * does. Thus:
2079 *
2080 * C = cholmod_triplet_to_sparse (T, 0, &Common) ;
2081 *
2082 * will return the matrix C = P*A*P'.
2083 *
2084 * Since the triplet matrix T is so simple to generate, it's quite easy
2085 * to remove entries that you do not want, prior to converting T to the
2086 * cholmod_sparse form. So if you include these entries in T, CHOLMOD
2087 * assumes that there must be a reason (such as the one above). Thus,
2088 * no entry in a triplet matrix is ever ignored.
2089 */
2090
2091 int itype ; /* CHOLMOD_LONG: i and j are SuiteSparse_long. Otherwise int */
2092 int xtype ; /* pattern, real, complex, or zomplex */
2093 int dtype ; /* x and z are double or float */
2094

2339 *
2340 * CHOLMOD_IS_NAN(x):
2341 * True if x is NaN. False otherwise. The commonly-existing isnan(x)
2342 * function could be used, but it's not in Kernighan & Ritchie 2nd edition
2343 * (ANSI C89). It may appear in <math.h>, but I'm not certain about
2344 * portability. The expression x != x is true if and only if x is NaN,
2345 * according to the IEEE 754 floating-point standard.
2346 *
2347 * CHOLMOD_IS_ZERO(x):
2348 * True if x is zero. False if x is nonzero, NaN, or +/- Inf.

2353 * This is (x != 0) if the compiler is IEEE 754 compliant.
2354 *
2355 * CHOLMOD_IS_LT_ZERO(x):
2356 * True if x is < zero or -Inf. False if x is >= 0, NaN, or +Inf.
2357 * This is (x < 0) if the compiler is IEEE 754 compliant.
2358 *
2359 * CHOLMOD_IS_GT_ZERO(x):
2360 * True if x is > zero or +Inf. False if x is <= 0, NaN, or -Inf.

2362 *
