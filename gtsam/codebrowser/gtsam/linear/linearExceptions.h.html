25 /**
26 Thrown when a linear system is ill-posed. The most common cause for this
27 error is having underconstrained variables. Mathematically, the system is
28 either underdetermined, or its quadratic error function is concave in some
29 directions.
30
31 Examples of situations causing this error are:
32 - A landmark observed by two cameras with a very small baseline will have
33 high uncertainty in its distance from the cameras but low uncertainty
34 in its bearing, creating a poorly-conditioned system.
35 - A landmark observed by only a single ProjectionFactor, RangeFactor, or
36 BearingFactor (the landmark is not completely constrained).
37 - An overall scale or rigid transformation ambiguity, for example missing
38 a prior or hard constraint on the first pose, or missing a scale
39 constraint between the first two cameras (in structure-from-motion).
40
41 Mathematically, the following conditions cause this problem:
42 - Underdetermined system: This occurs when the variables are not
43 completely constrained by factors. Even if all variables are involved
44 in factors, the variables can still be numerically underconstrained,
45 (for example, if a landmark is observed by only one ProjectionFactor).
46 Mathematically in this case, the rank of the linear Jacobian or Hessian
47 is less than the number of scalars in the system.
48 - Indefinite system: This condition occurs when the system Hessian is
49 indefinite, i.e. non-positive-semidefinite. Note that this condition
50 can also indicate an underdetermined system, but when solving with
51 Cholesky, accumulation of numerical errors can cause the system to
52 become indefinite (have some negative eigenvalues) even if it is in
53 reality positive semi-definite (has some near-zero positive
54 eigenvalues). Alternatively, if the system contains some
55 HessianFactor's with negative eigenvalues, these can create a truly
56 indefinite system, whose quadratic error function has negative
57 curvature in some directions.
58 - Poorly-conditioned system: A system that is positive-definite (has a
59 unique solution) but is numerically ill-conditioned can accumulate
60 numerical errors during solving that cause the system to become
61 indefinite later during the elimination process. Note that poorly-
62 conditioned systems will usually have inaccurate solutions, even if
63 they do not always trigger this error. Systems with almost-
64 unconstrained variables or vastly different measurement uncertainties
65 or variable units can be poorly-conditioned.
66
67 Resolving this problem:
68 - This exception contains the variable at which the problem was
69 discovered (IndeterminantLinearSystemException::nearbyVariable()).
70 Note, however, that this is not necessarily the variable where the
71 problem originates. For example, in the case that a prior on the
72 first camera was forgotten, it may only be another camera or landmark
73 where the problem can be detected. Where the problem is detected
74 depends on the graph structure and variable elimination ordering.
75 - MATLAB (or similar software) is a useful tool to diagnose this problem.
76 Use GaussianFactorGraph::sparseJacobian_(),
77 GaussianFactorGraph::sparseJacobian()
78 GaussianFactorGraph::denseJacobian(), and
79 GaussianFactorGraph::denseHessian() to output the linear graph in
80 matrix format. If using the MATLAB wrapper, the returned matrices can
81 be immediately inspected and analyzed using standard methods. If not
82 using the MATLAB wrapper, the output of these functions may be written
83 to text files and then loaded into MATLAB.
84 - When outputting linear graphs as Jacobian matrices, rows are ordered in
85 the same order as factors in the graph, which each factor occupying the
86 number of rows indicated by its JacobianFactor::rows() function. Each
87 column appears in elimination ordering, as in the Ordering class. Each
88 variable occupies the number of columns indicated by the
89 JacobianFactor::getDim() function.
90 - When outputting linear graphs as Hessian matrices, rows and columns are
91 ordered in elimination order and occupy scalars in the same way as
92 described for Jacobian columns in the previous bullet.
93 */
94 class GTSAM_EXPORT IndeterminantLinearSystemException : public ThreadsafeException<IndeterminantLinearSystemException> {
95 Key j_;
96 public:
97 IndeterminantLinearSystemException(Key j) noexcept : j_(j) {}
98 ~IndeterminantLinearSystemException() noexcept override {}
99 Key nearbyVariable() const { return j_; }
100 const char* what() const noexcept override;
101 };

104 /** An exception indicating that the noise model dimension passed into a
105 * JacobianFactor has a different dimensionality than the factor. */
106 class GTSAM_EXPORT InvalidNoiseModel : public ThreadsafeException<InvalidNoiseModel> {
107 public:
108 const DenseIndex factorDims; ///< The dimensionality of the factor
109 const DenseIndex noiseModelDims; ///< The dimensionality of the noise model
110
111 InvalidNoiseModel(DenseIndex factorDims, DenseIndex noiseModelDims) :
112 factorDims(factorDims), noiseModelDims(noiseModelDims) {}
113 ~InvalidNoiseModel() noexcept override {}
114
115 const char* what() const noexcept override;
116 };
117
118 /* ************************************************************************* */
119 /** An exception indicating that a matrix block passed into a
120 * JacobianFactor has a different dimensionality than the factor. */
121 class GTSAM_EXPORT InvalidMatrixBlock : public ThreadsafeException<InvalidMatrixBlock> {
122 public:
123 const DenseIndex factorRows; ///< The dimensionality of the factor
124 const DenseIndex blockRows; ///< The dimensionality of the noise model
125
126 InvalidMatrixBlock(DenseIndex factorRows, DenseIndex blockRows) :
127 factorRows(factorRows), blockRows(blockRows) {}
128 ~InvalidMatrixBlock() noexcept override {}
129
130 const char* what() const noexcept override;
131 };
